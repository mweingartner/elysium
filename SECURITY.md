# Security Policy

## What Elysium is (and isn't)

Elysium is a local macOS game with player-started LAN multiplayer session support. Local-network multiplayer is deliberately scoped to Bonjour discovery, Direct Connect, bounded handshakes, peer status, LAN chat, host-authoritative replication batches, and host-authorized gameplay intents. Clients can request actions with typed intents, but they cannot send raw save data, arbitrary commands, authoritative RPG state, or authoritative world state; the remote-player gameplay layer enforces permissions for build, container, crafting, template, command, AI, creative, dimension, respawn, death, and reconnect flows.

Current release properties:

- **No external service access.** The app has no telemetry, analytics, update checks, account service, NAT traversal, relay, or cloud multiplayer. The in-app network surfaces are player-started LAN multiplayer over Network.framework and the optional `/ai` command, which is hard-coded to local Ollama on `http://localhost:11434`; the `elysium update` shell command separately runs `git pull` on your own checkout.
- **No accounts, no credentials, no personal data.** Elysium stores worlds, settings, and keybinds under `~/Library/Application Support/Elysium/`.
- **No elevated privileges.** It's an ad-hoc-signed app running in a normal user session.

That makes the realistic threat model: **malicious files that you load into the game, malformed output from local tools you explicitly connect to, and malformed traffic from peers on a local network you choose to join or host.**

RPG local preferences use a closed ElysiumStorage component capability with exact schema markers, row/count/aggregate byte caps, strict `PBLQS1` decoding, domain-separated SHA-256 verification, immutable legacy-origin binding, stale-CAS rejection, and atomic deletion with the owning world. LAN-shaped identities cannot enter this local façade. The LAN-v6 client checkpoint schema and aggregate storage façade are compiled, but component bootstrap remains test-only and the production accessor is verify-only. ElysiumCore has no client checkpoint codec, candidate builder, `SaveDB` entry point, or façade acquisition: those remain blocked on the reviewed Phase-2.5 owner/request/send-state/disposition codecs and transition matrix, so protocol 5 cannot persist or publish pseudo-validated authority bytes. Host owner rows expose neither a writer nor a bootstrap. Component bootstrap accepts only complete legal prefixes and schema audit rejects partial components, alternate DDL, unexpected indexes, views, and user triggers.

Saved-world batch deletion is fail-closed against corrupted or racing local database state. Checked snapshots cap IDs at 256 UTF-8 bytes, rows at 4,096, aggregate raw row data at 67,108,864 bytes, request/receipt envelopes at 1,310,720 bytes, chunk rows at 1,048,576, and statement work at 53,248 through one overflow-checking production accumulator. Domain-separated SHA-256 binds exact IDs, JSON bytes, `lastPlayed` bits, collection order, requests, and receipts. The single transaction pre-counts and post-proves world, chunk, player, advancement, RPG preference, and RPG migration scopes; prepared delete statements are reset and clear-bound between IDs. Ambiguous continuation is a request-bound read-only authority: it never reissues deletion, exact-post proof retires only the selected RPG omission IDs, and exact-pre or unresolved proof preserves them and the global maintenance lease. Recovery uses a verify-only RPG-local schema audit; missing, partial, or corrupt component objects remain absent/corrupt and terminal rather than authorizing bootstrap or mutation. Compiler-enforced main-actor admission guards world transitions and LAN hosting. UI and Accessibility never expose raw IDs, digests, SQL, paths, corrupt values, underlying errors, control characters, Unicode format controls, or bidi direction controls, and revalidate screen generation, key-window membership, enabled state, and action identity before Press.

The RPG UI harness is detected before ordinary app construction and fails closed when its exact six-key

`O_EXCL|O_NOFOLLOW` and mode `0600`, and clean an incomplete reservation on termination. Semantic
output is sorted/bounded and excludes player, owner, credential, path, and raw notice payload data.

NPC barter treats saves, merchant state, and receipts as hostile inputs. Decoding caps payload and offer counts and validates registry ids, stack metadata/counts, stable ids, tiers, uses, XP, timers, and duplicates before secondary materialization or registry indexing. Opaque one-shot receipts bind runtime object identity, world/player/merchant attachment, reach/line of sight, merchant revision, offer digest, inventory digest, and a bounded nonce set; replay, stale state, overflow, and LAN-client calls fail before mutation. Payment and result insertion are simulated on a detached inventory, then inventory and merchant post-state publish before sound, particles, or advancement callbacks.

Worldgen planning never trusts partially generated target state as cross-chunk authority. The exact bounded base-terrain oracle and context-complete single-flight caches prevent seed/preset aliasing and unbounded duplicate work; candidate and piece caps constrain adversarial coordinates/settings. Village and dungeon planners validate before detached writes, dungeon block entities are checked against their owning cells before commit, and underwater selection is region-budgeted. Existing saved full chunks remain authoritative and receive no migration or automatic seam repair.

Checked player-row persistence exposes only absent/exact-digest expectations and conflict/committed
results. Its private authorizer reads the exact world parent and player row, permits only player
INSERT or JSON-only UPDATE, and returns redacted Core errors. Candidate encoding occurs before rank
12; stale writers, missing parents, world deletion, hostile row classes, and transaction faults
cannot turn the CAS into an unconditional overwrite.

Host runtime allocation/cap testing remains blocked on the future reviewed host parent schema and
coherent host façade; no dormant test façade widens that boundary.

Local verification scripts:

- `./scripts/security-scan.sh` checks source for unapproved network/process/dynamic-loading APIs and obvious secret material. Its bounded local-release checks reject production fixture/fault selectors, verify release entry-point ownership and modes, validate the exhaustive source-snapshot contract, and reject retired release-authority surfaces. It also runs the lexical SQLite boundary scanner, compiling its bypass fixture, rejecting SQLite/SQL/storage-capability escapes, comparing the SwiftPM source inventory and dependency direction, and byte-pinning ElysiumStorage's package/SPI symbol graph plus the exact two-file Core capability inventory. Network APIs are allowed only in the loopback Ollama client and the LAN transport adapter.
- `./scripts/verify-pack-assets.sh` verifies the bundled Faithful archive, including the `assets/minecraft/textures/` namespace Elysium uses for its default graphics.
- `./scripts/security-check-binary.sh /Applications/Elysium.app` verifies the app signature, bundle metadata, linked library paths, and network-related binary symbols/strings. It allows only the local Ollama URL, requires LAN privacy/Bonjour declarations before accepting Network.framework symbols, and rejects other URL literals.
- `./scripts/pipeline.sh` runs the full source-security, warning-free build, release-surface/binary, XCTest, golden, package, packaged-AppKit, install, and installed identity/codesign pipeline.

The unattended AppKit gate is deliberately fail closed. It requires the current unlocked console user plus
Accessibility/Input Monitoring trust, a regular hash-pinned release executable, strict ad-hoc
signing and resource sealing, and exact launched bundle/PID identity. It has zero general-pasteboard
or Paste activity. Deadlines, signal cleanup, redacted diagnostics, and release rehashing prevent an
adapter-only or stale installed binary from satisfying it. Production Paste adapter tests cover
readiness/owner/bounds/race behavior.

The pipeline binds all tracked and nonignored-untracked regular inputs at entry and revalidates them
after every stage. It rejects persistent additions, removals, replacements, mode changes, unsafe
links/types, and boundary-visible races. The current console user and workspace are trusted: an
exact same-user mutate-and-restore wholly between boundary scans and malicious local build plugins
are explicitly out of scope. The pipeline therefore does not claim concurrent-tamper resistance.
Executable SHA continuity remains mandatory from warning-free build through signed package,
packaged AppKit execution, installation, and final independent identity/signature verification.

## Reporting a vulnerability

If you find a way for a crafted save file or texture archive to do anything beyond crashing the game (memory corruption, code execution, file writes outside the support directory), please report it privately:

- **Email:** briangaoo2@gmail.com — subject line starting with `[elysium security]`
- Include: macOS version, a minimal reproducing file if possible, and what you observed.

Plain crashes / hangs from malformed files are ordinary bugs — file those as [regular GitHub issues](https://github.com/mweingartner/elysium/issues) with the offending file attached (the README lists what else to include). This is a beta; reports of every kind are incredibly welcome.

You can expect an acknowledgment within a few days. There's no bug bounty; you'll get credit in the changelog and my genuine thanks.

## Supported versions

Only the latest release is supported. There's no backporting; the fix ships in the next version.
